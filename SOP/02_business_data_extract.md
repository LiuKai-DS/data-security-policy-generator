# 第二阶段：业务数据提取（精细化运营）

## 核心思想

> **提炼企业实际业务数据，针对性监控。**

在MVP阶段基础上，通过数据分级分类、调研、业务梳理等方式，提炼出企业实际业务数据，并将其制定成安全防护策略。

### 第四章：通道一 — 公司名称联网检索

**触发条件**：用户提供了公司名称（SKILL.md ⑤A）

#### 检索策略

```
检索优先级（从高到低）：
1. 官方年报/公告（最权威）→ 主营业务章节、产品目录、组织架构
2. 公司官网"关于我们"/"业务介绍"  → 产品名称、系统名称
3. 监管公告/招股书/行政许可文件    → 合规关键数据字段
4. 行业媒体新闻（补充）            → 公司特有术语、行业热点词
```

#### 提取重点（按数据等级）

| L级别 | 提取重点 | 检索来源 |
|-------|---------|---------|
| **L4** | 核心产品名称、自研系统名称、核心业务线 | 年报"主营业务"章节 |
| **L3** | 客户群体类型、合作伙伴分类、核心合同类型 | 官网合作案例 |
| **L2** | 内部系统名称（OA/ERP/CRM命名）、业务编号规则 | 招聘JD/技术分享文章 |

#### 输出格式示例

```
【精准化关键字草稿 - 通道一：公司名称检索】
公司名称：××集团股份有限公司
检索时间：2026-04-22
信息来源：年报/官网/公告

L4 专属关键字（新增）：
  + ××超纯水制备工艺（主营产品）
  + ××智能运维平台（自研系统）
  + MNCS系统（生产管理系统，年报提及）

L3 专属关键字（新增）：
  + 华润×× 商标授权（年报授权关系）
  + KA客户协议（年报"大客户管理"章节）

L2 专属关键字（新增）：
  + 内控评价报告（年报合规章节）

→ 合并通用BU字典后共计关键字：XX条（较通用版新增XX条）
```

### 第五章：通道二 — 规章制度文档解析

**触发条件**：用户上传了企业规章制度文件（SKILL.md ⑤B）

#### 文档解析流程

```
Step 1：文档扫描
  - 读取所有可处理的文档段落
  - 识别文档结构（章节标题→数据分类层级）

Step 2：敏感实体识别
  目标词汇类型：
  ├── 数据对象名称（"客户档案"、"项目底稿"）
  ├── 系统/平台名称（"ERP系统"、"××运营平台"）
  ├── 业务术语（企业内部命名，如"商机"、"里程碑"）
  ├── 文件类型命名（"××类合同"、"X号保密文件"）
  └── 禁止行为描述（直接提取禁止外发的数据类型）

Step 3：级别自动映射
  规则：
  ├── 文档明确标注L4/L3/L2 → 直接引用
  ├── 文档中写明"核心商密/不得外传" → 标记为L4候选
  ├── 文档中写明"内部保密/仅内部" → 标记为L3候选
  └── 其余内部术语 → 标记为L2候选，由用户确认

Step 4：过滤与清洗
  - 自动过滤：员工姓名、身份证号、联系方式（不作为关键字）
  - 自动去重：与通用字典重叠的词汇标记"已覆盖"，不重复添加
  - 过窄词过滤：单字/过度通用词（如"数据"、"文件"）自动提示用户确认

Step 5：输出预览（先预览再确认）
```

#### 输出格式示例

```
【精准化关键字草稿 - 通道二：规章制度解析】
文档名称：××公司数据分类分级管理规范V2.0
文档级别：内部公开

从文档中提取到以下专属关键字：

L4（文档定义为"核心数据"）：
  + 经营战略规划书（第3.1条定义）
  + OAM系统数据库结构文件（第3.1条定义）
  [已覆盖] 高管薪酬（通用字典已有，无需新增）

L3（文档定义为"重要数据"）：
  + 年度预算执行报告（第3.2条定义）
  + 客户信用评估报告（第3.2条定义）

L2（文档定义为"内部数据"）：
  + 门禁系统日志（第3.3条定义）

⚠️ 待用户确认：
  - "项目资料"（第3.2条，范围较宽，建议细化为"项目投标资料"）
  - "ERP数据"（建议细化为具体模块名称）

→ 合并后关键字共计：XX条（较通用版新增XX条，与文档定义对齐）
```

### 第六章：双通道融合输出

**当 ⑤A 和 ⑤B 同时启用时**，按以下优先级融合：

```
优先级（高→低）：
① 规章制度文档中明确定义的级别  （最权威，直接采用）
② 联网检索到的公司专属业务词汇  （补充覆盖文档未提及的业务词）
③ 行业BU关键字字典（04/05）    （兜底覆盖）
```

**融合规则**：
- 同一词汇在不同来源有冲突时，以**文档定义级别为准**（通道二优先）
- 通道一新增的词汇如在通道二未被定义，保留但标注"来源：公开信息，建议与业务部门确认"
- 最终输出前，**必须向用户提供预览草稿**，确认无误后再写入策略表

## 业务数据提取方法

### 一、提炼业务数据的方法

**1. 数据分级分类调研**
- 访谈各部门负责人，梳理本部门涉及的核心业务数据
- 识别数据产生、存储、流转、消亡的全生命周期
- 确定每类数据的敏感级别（L1-L4）

**2. 关键字来源提取**
- 从业务系统中提取敏感字段名称（如CRM系统中的"客户名称"、"商机金额"）
- 从历史外发邮件中提取高频敏感词
- 从OA/公文系统中提取业务流程术语
- 从合同管理系统中提取合同编号规则、金额格式

**3. 数据字典建立**
- 建立企业级敏感数据字典
- 按部门分类，每类数据包含：数据名称、数据级别、关键字来源、常见外发场景

## 业务数据提取常见误区

| 误区 | 正确做法 |
|------|---------|
| 只提取明显敏感词 | 结合业务场景，提取"看起来普通但业务敏感的词" |
| 照搬其他企业关键字 | 必须基于本企业实际业务数据定制 |
| 关键字过于宽泛 | 如"数据"二字太宽泛，应精确到"客户数据"、"生产数据" |
| 忽视数据级别 | 同一类数据可能涉及多个级别，需分开定义 |
| 一次性提取完毕 | 业务数据字典需持续迭代，建议每季度更新 |

## 第二阶段实施建议

1. **先建立分级分类框架**：明确L1-L4各级定义，统一口径
2. **从高敏感部门入手**：先做合规监督部、财务管理部等核心部门
3. **交叉验证关键字**：通过历史外发日志验证提取的关键字覆盖率
4. **建立数据字典**：形成可维护、可迭代的企业级敏感数据字典
